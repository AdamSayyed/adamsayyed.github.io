High-Performance 3D Heat Diffusion Engine

MPI, OpenMP, CUDA, Slurm

View Project on GitHub

1. Introduction & Goals

This project focused on building a high-performance scientific simulation for 3D heat diffusion and systematically optimizing it using multiple HPC techniques:

  • Shared-memory parallelism (OpenMP)
  • Distributed-memory parallelism (MPI)
  • Hybrid MPI + OpenMP execution
  • Cluster-level scaling via Slurm
  • GPU acceleration using CUDA
  • Performance profiling and strong scaling analysis

The objective was to understand how memory bandwidth, cache behavior, and communication overhead shape performance in real scientific workloads.

2. Baseline Implementation (CPU, MPI, OpenMP)

A finite-difference stencil solver was implemented to update a 3D grid over time:

  • MPI handled domain decomposition and halo exchange
  • OpenMP executed updates within each sub-domain
  • A custom profiler measured per-rank wall clock times
  • I/O module wrote checkpoint files every 10 timesteps

Baseline Performance (200³ Grid)

MPI Ranks Time (s)
1 10.84
2 3.41
4 1.62
8 0.94

Results show near-perfect speedup up to 4 ranks and saturation by 8 ranks due to memory bandwidth limits.

3. Optimization 1 — Cache Blocking (Tiling)

Implemented explicit cache tiling to improve data locality in solver_cpu.cpp.

Outcome:

  • Speedup was small (~1.1×)
  • Memory bandwidth remained the main bottleneck
  • MPI’s natural sub-domain decomposition effectively acted as a better "implicit" cache tiling method

Key Insight: Shared-memory tiling is overshadowed by distributed-memory decomposition for this memory-bound stencil.

4. Optimization 2 — Multi-Node Scaling

Used Slurm to run the simulation across multiple compute nodes.

Small Grid (200³)

No speedup — problem fits into the CPU caches, and network overhead dominates.

Large Grid (400³)

When data exceeded LLC cache capacity:

  • Single node: 58.8s
  • Two nodes: 7.3s
  • Speedup: 8.0× (perfect linear scaling)

Key Insight: Large grids shift bottlenecks from cache to RAM, allowing multi-node execution to multiply available memory bandwidth.

5. Optimization 3 — GPU Acceleration (CUDA)

Implemented a CUDA kernel in solver_cuda.cu for GPU execution.

Naive GPU Version:

  • Full grid copied Host → Device → Host every timestep
  • Strong PCIe bottleneck

Performance (400³)

Platform Time (s)
CPU (8 cores) 58.8
GPU (naive) 34.2

Speedup: ~1.7×
Potential: Keeping the grid resident on GPU memory could yield 10×–50×, similar to production CFD/HPC GPU solvers.

6. Comprehensive Analysis

Strategy Speedup Bottleneck Notes
MPI Only 6×–8× Memory bandwidth Great intra-node scaling
Tiling 1.1× Memory bandwidth Limited impact
Multi-Node MPI 8× Network latency Perfect scaling at large N
GPU (Naive) 1.7× PCIe bandwidth Huge potential with redesign

7. Key Takeaways

  • Memory bandwidth is the primary limiter for stencil computations.
  • Hybrid MPI+OpenMP is optimal for node-level scaling.
  • GPU compute is very fast, but PCIe transfers destroy performance unless minimized.
  • Multi-node runs excel when the problem outgrows a single node’s cache & RAM bandwidth.
